# Project Intent

Status: draft / active / stale
Last reviewed:

Use when planning, exploring direction, making ambiguous scope decisions, or
reviewing whether work still serves the project's stated value. Do not route
ordinary implementation agents here by default; route them here only when the
work brief names this document as needed context.

This document is human-owned. Agents may help clarify, challenge, summarize, or
propose edits, but humans own the project's intent and any accepted changes to
it.

Keep this short. Prefer fewer than 50 lines, and do not exceed 100 lines. Move
details into PRDs, ADRs, roadmap docs, domain docs, or work briefs.

Expected change cadence: early project intent may change frequently while the
team learns. Mature project intent should change deliberately and more slowly.

## Who This Serves

Who are the users, maintainers, operators, stakeholders, or systems this project
exists to serve?

## Problem

What problem is this project trying to solve?

## Value Proposition

What value should this project create for the people or systems it serves?

## What Good Looks Like

How will humans recognize that the project is succeeding?

## Current Direction

What is the current high-level direction for product, design, architecture,
domain model, or operational shape?

## Boundaries

Optional. Name only the few boundaries that prevent repeated scope confusion.

## Review Triggers

Revisit this document when repeated work conflicts with it, project direction
changes, or humans and agents keep making inconsistent value judgments.
